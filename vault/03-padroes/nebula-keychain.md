# Keychain — NebulaSecureStore + NebulaKeychain (Wave N9 — shipped)

The secure-storage half of the app-readiness surface ([[nebula-app-readiness-research]] N9, gated by the resolved "non-UI system frameworks" scope Q). A `Sendable` key-value **port** (`NebulaSecureStore`) plus a concrete stateless `final class` façade over the Security.framework `SecItem*` C API (`NebulaKeychain`). Source of truth = `Sources/Nebula/Architecture/Keychain/`; this note is synthesis. Full API verification + the N10 auth-interceptor design live in the research note [[nebula-keychain-auth]].

## Design decisions

- **`NebulaKeychain` is Mutex-free — the key refinement vs. the vault note's "Mutex-façade" framing.** `NebulaDefaults` wraps `UserDefaults` in a `Mutex<UserDefaults>` because `UserDefaults` is a **non-`Sendable` object** that must be region-isolated. The Keychain `SecItem*` C API is different: there is **no Swift object to hold** — they are thread-safe free functions parameterized by a fresh query dictionary per call. So `NebulaKeychain` is a **stateless `final class`** holding an immutable `let config: NebulaKeychainConfig` (a `Sendable` value type). A `final class` with a single `let` `Sendable` property **derives `Sendable`** with no `Mutex` and no `@unchecked` — the `NebulaError.Box` precedent (`Sources/Nebula/Errors/NebulaError.swift:95-102`), NOT the `NebulaDefaults` `Mutex` precedent. A single instance is safe to share across tasks (the 50-task concurrency test proves it).
- **`OSStatus` is stored, not boxed.** `OSStatus` is `Int32` (trivially `Sendable`). It lives on the error struct as `var status: OSStatus`, not in `underlying: NebulaError.Box?` (which is reserved for a nested `NebulaError`). This mirrors the `NWError`-folded-into-message idiom at `NebulaHTTPServer.swift:104` — lossy, Sendable-only fragments kept. `toNebulaError` writes `metadata["NebulaOSStatus"] = String(status)` and `code.code = Int(status)`.
- **`Kind.cocoa` is the bridge target for `errSec*` codes — no new `NebulaError.Kind` case.** `errSec*` OSStatus codes are CoreFoundation/OSStatus failures → `.cocoa` (the `CocoaError` bucket, per `NebulaError.swift:64` + the `CocoaError` bridge at `NebulaError+Mapping.swift:117-121`). `itemNotFound`/`cancelled`/`unknown` → `.unknown` (the not-found distinction lives in the per-layer `Kind`, like `NebulaRepositoryError.Kind.notFound`). Picked up by the `as? NebulaFailure` branch in `NebulaError(error:)` (`NebulaError+Mapping.swift:145-151`) — no dispatch change.
- **Fresh query dict per call.** Every `SecItem*` call builds a fresh `[String: Any]` from `baseQuery(account:)` (the Keychain best practice — never mutate a shared dict). `baseQuery` carries the immutable match attributes: `kSecClassGenericPassword` + `kSecAttrService` + `kSecAttrAccount` + optional `kSecAttrAccessGroup` + optional `kSecUseDataProtectionKeychain`.
- **`SecItemUpdate` over delete-re-add.** `setData(_:forKey:)` calls `SecItemUpdate` first (attributes-to-update = `kSecValueData`), falling back to `SecItemAdd` only on `errSecItemNotFound`. This preserves access control and avoids a re-prompt (the vault-note rule) — delete-then-add would discard the item's `kSecAttrAccessible` / access control. `errSecDuplicateItem` on the add path is a TOCTOU race (another writer added between the update-not-found and the add) — thrown honestly.
- **`errSecInteractionNotAllowed` is non-destructive.** The device is locked, not the item missing. `remove` throws on it (the `SecItemDelete` already failed — nothing was deleted); `setData` throws on it (no `remove` is called); `data` throws on it. No path deletes on this error.
- **`remove` is idempotent.** `errSecItemNotFound` from `SecItemDelete` is a no-op success (absent is not an error for a delete).
- **`kSecUseDataProtectionKeychain = true` is the production default** (the modern, cross-platform, no-prompt, sandbox-friendly keychain; iOS-style on macOS). On the unsigned `swift test` host (`CODE_SIGNING_ALLOWED=NO`) this may return `errSecMissingEntitlement`, so the **integration tests override to `false`** (login-keychain path); the production default stays `true`. Flagged as an implementation-time verification with a fallback; verified `false` works on the macOS host.
- **`NebulaSecureStore` is a distinct port from `NebulaPreferences`.** Same three byte-level signatures, but `throws` and a separate seam. A Keychain conformer must not be a drop-in for a prefs store — secrets and user-tunable preferences have different threat models. An app injects a secure store and a preferences store as separate seams.
- **Per-call `JSONEncoder`/`JSONDecoder`** in the Codable bridge (decoupled from `NebulaJSONEncoder`/`NebulaJSONDecoder`), matching `NebulaPreferences.swift:58,69` — a secret shouldn't inherit the gateway's date-encoding strategy.

## TDD fit (Wave N9)

- Each integration test uses a unique `kSecAttrService` (`#function` + `UUID`) and `defer`-cleans by service — no cross-test pollution. The integration suite is `@Suite(.serialized)` to avoid `errSecDuplicateItem` races against the real Keychain.
- `InMemorySecureStore` (`final class`, `Mutex<[String: Data]>`) is a non-Keychain conformer; its tests prove the `Codable`/`RawRepresentable` default extension is reusable on any conformer, not just `NebulaKeychain`. The existential test holds two impls in `[NebulaSecureStore]`.
- The concurrent-access test runs 50 child `Task`s through one shared `NebulaKeychain`, each writing a unique key and reading it back; the `SecItem*` C API is thread-safe, so no `Mutex` is needed — a data race would crash or return a wrong value. Child tasks catch internally so the `addTask` closure stays non-throwing.
- The `errSecInteractionNotAllowed` mapping is asserted via the error struct directly (the live API can't simulate a locked device without a locked device).

## Notes / guardrails

- `import Security` needs **no `Package.swift` change** — system frameworks are auto-linked on Apple SDKs (the `import CryptoKit` precedent at `NebulaHashAlgorithm.swift:26`). `dependencies: []` stays pristine.
- `SecItem*` / `kSecAttrAccessible*` are 5-platform (no `API_UNAVAILABLE` for tvOS/watchOS/visionOS) → **no `@available` gates** on `NebulaKeychain`. Biometry-gated items (`SecAccessControlCreateWithFlags` + `LAContext`) are **deferred** — `LAContext` is `API_UNAVAILABLE(tvOS)` at the class level → can't be 5-platform (a platform carve-out, not a scope question). N9 uses `kSecAttrAccessible` only.
- **Lean v0.6 surface**: no key-namespace policy, no access-control-create (biometry), no keychain-sharing helper beyond the `accessGroup` seam, no change observation. Those wait for a use (N10 / a biometry wave).
- No new `NebulaError.Kind` case (closed-enum rule); `NebulaKeychainError` bridges through the existing `as? NebulaFailure` dispatch.

## Test-stability fix surfaced during N9 verification

The full suite exposed a **pre-existing race** in `MeasureTests.swift` `NebulaMeasureConfigTests`: the two process-wide-accessor tests (`getReturnsDefaultBeforeAnySet` / `setReplacesCurrentConfiguration`) race on the shared `NebulaMeasureConfig` under Swift Testing's default parallelism — one test's `defer { set(.default) }` can land between the other's `set(custom)` and its `get().bench()`, so `bench` fires the default no-op handler and `captured.count == 0` ≠ 1. The N9 suites shifted the parallel schedule enough to surface it. Fix: `@Suite("…", .serialized)` on that suite (the two accessor tests are the only touchers of the process-wide accessor, verified by grep). One-line, clearly-correct; recorded in the N9 ADR.
