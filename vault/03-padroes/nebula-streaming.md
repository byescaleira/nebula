# Nebula — Streaming (SSE + WebSocket) (shipped)

> Shipped note for the streaming dimension of [[nebula-app-readiness-research]] (wave N17b, a segunda sub-wave do N17 split). Source of truth: `Sources/Nebula/Architecture/Network/Streaming/` (7 arquivos) + `Tests/NebulaTests/ArchitectureSSETests.swift` + `Tests/NebulaTests/ArchitectureWebSocketTests.swift` + `Sources/Nebula/Nebula.docc/ArchitectureStreaming.md`. Parent research: [[nebula-network-hardening]]. Reusa o pinning do [[nebula-ssl-pinning]] (N17a) via composição/forwarding — **zero mudança no N17a**.

## O que shipou (Nebula 0.15.0)

Duas superfícies **aditivas** sobre o gateway HTTP bufferizado existente (`NebulaHTTPGateway.send` retorna `NebulaHTTPResponse` com `body: Data` — verificado; não há método streaming em `NebulaHTTPClient`). O gateway **não foi tocado**; o pinning do N17a é reusado via composição/forwarding.

### SSE — Server-Sent Events (2 arquivos)

- **`NebulaSSEEvent`** — `Sendable`/`Equatable`/`Hashable` struct: `id: String?` (último `id:`, carregado entre dispatches → header `Last-Event-ID` no reconnect), `event: String` (default `"message"`), `data: String` (multi-line `data:` joined com `\n`, trailing newline stripped per WHATWG), `retry: Int?` (ms). Sendable derived.
- **`NebulaSSEParser`** — `internal struct` **pura** state machine (o testable seam). `@discardableResult mutating func feed(_ line: String) -> NebulaSSEEvent?`. Sem `URLSession`, sem `@Sendable` closure — unit-tested com canned `[String]`. WHATWG: `:` comment ignorado; `field: value` split no primeiro `:`, um espaço após o `:` stripped; linha sem `:` → field com value vazio; `event` seta tipo; `data` append `value + "\n"`; `id` seta lastEventID (rejeitado se contém U+0000 NULL); `retry` parseia int não-negativo (lossy); fields desconhecidos ignorados; blank line despacha (quando `data` não-vazio), resetando `data`/`event` mas **preservando** `lastEventID`/`retry`. **Desvio documentado**: blank line só com `id:`/`event:`/`retry:` (sem `data:`) **não** despacha evento vazio espúrio (heartbeats são `:` comments, não blank lines). Trailing `\r` stripped defensivamente.
- **`NebulaSSEConfiguration`** — `Sendable` (**NÃO `Equatable`** — o `sleeper` é `@Sendable`, mirror do flavor not-`Equatable` do `NebulaLogConfiguration`). **Per-call**, não process-wide → **sem accessor `Mutex`** (diferente do logging/measurement/error). Fields: `reconnect: Bool = true`, `maxReconnectAttempts: Int = 5`, `reconnectDelay: Duration = .seconds(3)`, `sleeper: @Sendable (Duration) async throws -> Void = NebulaRetry.defaultSleep`. Fluent `.withReconnect/.withMaxReconnectAttempts/.withReconnectDelay/.withSleeper`. `default = .init()`.
- **`NebulaSSEEventStream`** — `enum` namespace (sem instâncias). `events(for:session:configuration:) -> AsyncThrowingStream<NebulaSSEEvent, any Error>` — o **concrete return type mandado pelo CLAUDE.md** (`some AsyncSequence` é ilegal num protocol requirement; a stream é retornada de um `static func`). Internamente itera o **raw `URLSession.AsyncBytes` byte stream** formando linhas manualmente — **NÃO `.lines`**: `AsyncLineSequence` skipa linhas vazias, mas SSE despacha no **blank line**, então `.lines` droparia toda fronteira de dispatch. Um `\n` (0x0A) termina a linha; buffer vazio no `\n` é o blank line que despacha. Flush de trailing partial line no fim do stream.
- **`NebulaSSEError`** — open-struct error mirroring `NebulaHTTPServerError`/`NebulaSSLPinningError`. `Kind` presets: `connect-failed`/`parse-failed`/`reconnect-exhausted`/`cancelled`/`unknown`. `coarseKind`: `.network` para connect/parse/reconnectExhausted, `.unknown` para cancelled/unknown (`default → .unknown`). Domain `"Nebula.NebulaSSEError"`. Factory statics. **Nenhum caso novo em `NebulaError.Kind`**.

### WebSocket — 5 arquivos

- **`NebulaWebSocketMessage`** — `Sendable`/`Equatable`/`Hashable` enum: `case data(Data)` / `case string(String)` — mirror do `URLSessionWebSocketTask.Message` como tipo **Nebula-owned** (o idiom é nunca expor o enum nested da Apple num port Nebula). `init?(_ message: URLSessionWebSocketTask.Message)` **failable** — `URLSessionWebSocketTask.Message` é non-frozen, então `@unknown default: return nil` (future case → `nil`, surfaced como `NebulaWebSocketError.unknown` pela façade). `var rawMessage: URLSessionWebSocketTask.Message`.
- **`NebulaWebSocketClient`** — `public protocol NebulaWebSocketClient: Sendable` (o port, mirror do `NebulaHTTPClient`, **sem** herança `NebulaGateway` — WebSocket é seu próprio axis, não request/response). Requirements: `send`/`receive`/`sendPing` (`async throws`), `close(with:reason:)` (`async`, non-throwing — o `cancel(with:reason:)` é sync non-throwing), `closeCode: URLSessionWebSocketTask.CloseCode?` (`.invalid → nil`), `maximumMessageSize: Int { get set }`.
- **`NebulaURLSessionWebSocket`** — `public final class NebulaURLSessionWebSocket: NebulaWebSocketClient, Sendable` façade sobre `URLSessionWebSocketTask`. Stored props: `let pinned: NebulaPinnedWebSocketSession` (ownership anchor), `let task: URLSessionWebSocketTask`, `let logger: NebulaLogger?`. **Sendable derived** — `URLSessionWebSocketTask` é `NS_SWIFT_SENDABLE` (`NSURLSession.h:1121` — **correção do research note anterior que dizia "NOT Sendable", falso neste SDK**); o `let task` deriva Sendable; **sem `@unchecked`**. `send`/`receive`/`sendPing` bridgeiam para os async do task; `receive()` faz `guard let message = NebulaWebSocketMessage(raw) else { throw .unknown }`. `sendPing` wrapa `task.sendPing` (completion-handler form, **não há overlay async** — verificado no swiftinterface) num `withCheckedThrowingContinuation`. `close(with:reason:)` → `task.cancel(with:reason:)`. Testable seams `internal`: `mapSend`/`mapReceive`/`mapPing`/`isCancellation`/`box`. `mapReceive` checa `task.closeCode != .invalid` → `.closed(code:reason:)`.
- **`NebulaWebSocketSessionDelegate`** — `public final class : NSObject, URLSessionWebSocketDelegate, Sendable` — UM objeto por sessão que faz pinning (via **composição** com o N17a `NebulaURLSessionDelegate`) AND surfaceia o lifecycle WebSocket (`didOpenWithProtocol`/`didCloseWith`). Stored props (todos `let`): `pinningDelegate: NebulaURLSessionDelegate`, `logger: NebulaLogger?`, `onOpen: @Sendable (String?) -> Void`, `onClose: @Sendable (URLSessionWebSocketTask.CloseCode, Data?) -> Void`. **Sendable derived** — todos `let` de tipo Sendable; `URLSessionWebSocketDelegate` é `@objc` protocol **NÃO** `NS_SWIFT_SENDABLE`, mas conformância a um `@objc` non-Sendable não bloqueia derived `Sendable` numa `final class` com props `let` Sendable — analogia `NebulaUNNotificationCenter`. **Probe EXIT=0** (Xcode 27 Beta 3 SDK, `swiftc -typecheck -swift-version 6 -strict-concurrency=complete -warnings-as-errors`). **Sem `@unchecked`.** O método auth-challenge **forwards** para `pinningDelegate.urlSession(session, didReceive: challenge, completionHandler:)` (púbico + callable) — **zero mudança no N17a, zero duplicação de lógica de pinning**, **sem `import Security`** (a eval `Sec*` vive no delegate N17a; esta classe só forwarda).
- **`NebulaWebSocketSession`** + **`NebulaPinnedWebSocketSession`** — `public struct NebulaPinnedWebSocketSession: Sendable { session; delegate }` (ownership explícita — `URLSession` **não** retém seu delegate, same footgun do `NebulaPinnedSession`). `enum NebulaWebSocketSession` com `pinned(by:configuration:logger:onOpen:onClose:) -> NebulaPinnedWebSocketSession` (compõe `NebulaURLSessionDelegate` dentro do `NebulaWebSocketSessionDelegate`) + `open(url:protocols:using:logger:) -> NebulaURLSessionWebSocket` + `open(request:using:logger:)` (cria o task via `session.webSocketTask(with:)`, `.resume()`, retorna façade retendo o pinned — o entry point ergonômico).
- **`NebulaWebSocketError`** — open-struct error mirroring `NebulaSSEError`. `Kind` presets: `connect-failed`/`send-failed`/`receive-failed`/`ping-failed`/`closed`/`cancelled`/`unknown`. `closed(code:reason:)` carrega o `WebSocketCloseCode` em metadata + reason em message. `coarseKind`: `.network` para connect/send/receive/ping/closed, `.unknown` para cancelled/unknown. Domain `"Nebula.NebulaWebSocketError"`. **Nenhum caso novo em `NebulaError.Kind`**.

## Ground truth verificado (empírico — Xcode 27.0.0 Beta 3 SDK / Swift 6.4; `.swiftinterface` + `NSURLSession.h`, citado por file:line)

**Todos os APIs de streaming estão ABAIXO do floor `.v26` em toda plataforma → NENHUM `@available` gate em N17b** (mesmo que N17a).

- `URLSession.bytes(for:)` — `Foundation.swiftinterface:16692` (`macOS 12 / iOS 15 / watchOS 8 / tvOS 15 / visionOS 1.0+`). `AsyncBytes: Sendable` (`:16701`).
- `AsyncThrowingStream` — `_Concurrency.swiftinterface:1396/1400` (`macOS 10.15 / iOS 13 / watchOS 6 / tvOS 13`). `Continuation` Sendable (`:1401`). `onTermination` (`:1419`, `(@Sendable (Termination) -> Void)?`, abaixo do floor).
- `URLSessionWebSocketTask` (class) — `NSURLSession.h:1121` — **`NS_SWIFT_SENDABLE`** → **a classe É `Sendable` em Swift** (correção load-bearing: o research note anterior que dizia "NOT Sendable" era FALSO neste SDK). `init`/`new` `NS_UNAVAILABLE` (`:1184`) → criado só via `URLSession.webSocketTask(with:)`.
- `URLSessionWebSocketTask.Message` — overlay `Foundation.swiftinterface:16615` `enum Message : Sendable { case data(Data); case string(String) }` (Sendable, **NÃO `Equatable`** — por isso o bridge é testado via round-trip + pattern match, não `==`).
- async `send(_:)`/`receive()` — overlay `:16622/16635`. `sendPing` — `NSURLSession.h:1154` completion-handler form only (**sem overlay async** → `withCheckedThrowingContinuation`). `cancel(with:reason:)` — `:1163` (sync, non-throwing; o nome Swift é `cancel(with:reason:)`, NÃO `close`). `maximumMessageSize`/`closeCode`/`closeReason` (`:1169/1175/1181`).
- `URLSessionWebSocketDelegate` — `NSURLSession.h:1922-1939` `@objc protocol <NSURLSessionTaskDelegate>`; `@optional urlSession(_:webSocketTask:didOpenWithProtocol:)` + `urlSession(_:webSocketTask:didCloseWith:reason:)` (Swift label `didCloseWith` com internal `closeCode` — o importer trimma "Code" do label; o `@objc` selector re-expande para `didCloseWithCode:reason:`). **NÃO `NS_SWIFT_SENDABLE`** → não bloqueia derived Sendable (analogia `NebulaUNNotificationCenter`). Delegate set no **session init** (`URLSession(configuration:delegate:delegateQueue:)`, `:255`); o task **não tem prop `delegate`**.
- `NWProtocolWebSocket.Options` (Network) — `autoReplyPing: Bool` (auto-responde pings, NÃO `autoReply`), `setSubprotocols(_:)`, `setClientRequestHandler(_:handler:)` (o server-side handshake — retorna `Response(status: .accept, subprotocol:, additionalHeaders:)`). `NWConnection.send(content:contentContext:isComplete:completion:)` (label `contentContext`, NÃO `context`; `completion: SendCompletion = .contentProcessed { }`) — o echo `send` **precisa** `NWProtocolWebSocket.Metadata(opcode:)` no `ContentContext` ou emite frame inválido (bug real encontrado e corrigido no echo server do teste). `connection.receiveMessage { data, context, isComplete, error in }` — `context.protocolMetadata` contém o `NWProtocolWebSocket.Metadata` com o opcode recebido (mirror para o echo).

## Por que o reconnect loop do SSE é custom, não `NebulaRetry.withPolicy`

O request **muta por tentativa** (o header `Last-Event-ID` avança com o cursor), e o `operation` do `withPolicy` é uma closure nullary que não pode mutar o request entre tentativas. O loop custom mirroria o contract de cancellation do `withPolicy` — cancellation é honrada imediatamente e **nunca retryed**.

## Semântica de cancellation do `AsyncThrowingStream` (correção load-bearing)

O research/plan original dizia "CancellationError rethrown ao consumer". **Incorreto para o consumer-facing `AsyncThrowingStream`**, verificado empiricamente (`debug_cancel.swift`): quando o consumer cancela seu `Task`, `AsyncThrowingStream.Iterator.next()` retorna `nil` (o `for try await` termina **normalmente**, NÃO throwa `CancellationError`), E `onTermination` fires com `.cancelled`. O consumer nunca observa o `finish(throwing: CancellationError())` interno — esse finish existe para tear down o `bytes(for:)` task internamente via `onTermination → loop.cancel()` (o loop interno observa a cancellation mid-op e finisha), mas acontece DEPOIS do consumer já ter terminado normally. O teste `cancellationEndsTheStreamGracefully` afirma terminação graciosa (hanging connection → 0 events, `task.value` retorna promptly, sem hang, sem retry storm) — NÃO `throws: CancellationError`. **DocC + este note corrigidos** para refletir isso.

## Composição/forwarding do pinning (reuso N17a)

`NebulaWebSocketSessionDelegate.urlSession(_:didReceive:completionHandler:)` é uma linha: `pinningDelegate.urlSession(session, didReceive: challenge, completionHandler: completionHandler)`. **Zero mudança no N17a, zero duplicação de lógica de pinning.** O pinning rides o delegate da sessão — o auth-challenge method é em `URLSessionDelegate`, que `URLSessionWebSocketDelegate` herda, então o **mesmo** delegate de sessão handleia pinning de HTTP(SSE) E WebSocket. `import Security` só no delegate N17a (a eval `Sec*`); o delegate WebSocket só forwarda (sem `import Security`).

## Assimetria de testabilidade

- **SSE via `URLProtocol`** — `URLProtocol` **intercepta** `URLSession.bytes(for:)` data tasks (confirmado via `debug_bytes.swift`); um `NebulaSSETestProtocol` serve um body SSE canned, com um flag `hang` (header enviado, body nunca → `bytes(for:)` bloqueia → teste de cancellation). `@Suite(.serialized)` (estado process-wide).
- **WebSocket via `NWListener` echo server** — `URLProtocol` **NÃO** intercepta `URLSessionWebSocketTask`, então o live round-trip precisa de um listener real (`Network.framework` admissível, precedente `NebulaHTTPServer`). O echo server handleia o handshake via `NWProtocolWebSocket.Options.setClientRequestHandler` + `autoReplyPing`, e echoa cada mensagem com o **mesmo opcode** (o echo `send` carrega `NWProtocolWebSocket.Metadata(opcode:)` no `ContentContext` — sem isso o frame é inválido e o client receive falha com "Socket is not connected"; bug real encontrado e corrigido). A lógica pura (parser, message bridge, error mapping, delegate lifecycle + auth-challenge forwarding, session builder) é unit-tested sem socket live.
