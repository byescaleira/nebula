It is the foundation counterpart to the Cosmos design-system string helpers, minus SwiftUI.\n\nGround truth was verified against the installed Xcode 27 Beta 3 / Swift 6.4 toolchain — the `Foundation.swiftmodule/arm64e-apple-macos.swiftinterface` and the `NSRegularExpression.h` ObjC header — and re-checked against Apple developer docs. This note was adversarially reviewed; corrections are inline.\n\n## Availability verdict (all 5 platforms, .v26 floor)\n\n| Surface | iOS | macOS | tvOS | watchOS | visionOS | vs. Nebula 26 floor above-floor base64 options as `@available(iOS 26.4, *)` overloads.\n- `Sources/Nebula/Extensions/String/NebulaString+Detection.swift` — `dataDetectedEntities(types:)`, `urls()`, `firstURL()` over a `Mutex<NSDataDetector?>`-cached `NSDataDetector`; prefer `matches(in:options:range:)` over the block `enumerateMatches`.\n- `Sources/Nebula/Extensions/String/NebulaString+Regex.swift` — thin forwarders (`contains(regex:)`, `replacing(regex:with:)`, `split(by:)`); the stdlib algorithms are already on `String`, so no re-wrapping of `firstMatch/wholeMatch/matches`.\n- `Sources/Nebula/Extensions/String/NebulaString+Localized.swift` — `NebulaStringLocalization` Sendable struct.\n- `Sources/Nebula/Extensions/String/NebulaString+AttributedString.swift` — `toAttributedString(_:)` Foundation-scope builder (Foundation ships `AttributeScopes.FoundationAttributes` without UIKit/SwiftUI).\n- `Sources/Nebula/Extensions/Regex/NebulaRegex.swift` — `NebulaRegex<Output>: Sendable where Output: Sendable`.\n- `Sources/Nebula/Extensions/Regex/NebulaRegexPatterns.swift` — curated `Regex` literals (structured tokens only; The `.string(_:)` implementation should reuse the `.lproj` sub-bundle + `.xcstrings` JSON fallback resolution from `CosmosLocalizationConfiguration` (see [[nebula-string-extensions]] sibling), because `String(localized:bundle:locale:)` does not reliably honor `locale` for lookup across all bundle layouts/OS versions.\n\n```swift\npublic struct NebulaRegex<Output>: Sendable where Output: Sendable {\n    public let regex: Regex<Output>\n    public init(_ regex: Regex<Output>) { self.regex = regex }\n    public init(@RegexComponentBuilder _ build: () -> Regex<Output>) { self.regex = build() }\n    public func firstMatch(in input: some StringProtocol) -> Regex<Output>.Match? { input.firstMatch(of: regex) }\n    public func wholeMatch(_ input: some StringProtocol) -> Regex<Output>.Match? { input.wholeMatch(of: regex) }\n    public func matches(in input: some StringProtocol) -> [Regex<Output>.Match] { Array(input.matches(of: regex)) }\n    public func contains(_ input: some StringProtocol) -> Bool { input.contains(regex) }\n}\n```\n\n`Regex<Output>` is `Sendable` when `Output: Sendable`, so `NebulaRegex<Output>` is `Sendable` under the same conditional constraint — a blanket conformance would be unsound for non-Sendable Output. The `@RegexComponentBuilder` closure is invoked once at init to produce an immutable `Regex` value — no closure is retained, so there is no `@Sendable`-closure storage concern.\n\n## Apple patterns adopted\n\n- Modern Swift Regex (literals + RegexBuilder DSL + `firstMatch(of:)`/`wholeMatch(of:)`/`matches(of:)`/`replacing(_:with:)`/`split(separator:)`/`trimmingPrefix`) over `NSRegularExpression` and hand-rolled scanning — per WWDC22 [Meet Swift Regex](https://developer.apple.com/videos/play/wwdc2022/110357/) / [Beyond the basics](https://developer.apple.com/videos/play/wwdc2022/110358/).\n- `String(localized:)` + `LocalizedStringResource` + `AttributedString(localized:)` (String Catalog stack, iOS 15 base / iOS 16 options) instead of `NSLocalizedString`; mirror `CosmosLocalizationConfiguration` deterministic-locale resolution.\n- `NSDataDetector` only for natural-language detection (not validation), per Apple's [NSRegularExpression.h](https://developer.apple.com/documentation/foundation/nsdatadetector) note; cache the instance (init is expensive, instances thread-safe post-init) behind a `Mutex<NSDataDetector?>` one-time init (Synchronization.Mutex is iOS 18/macOS 15/tvOS 18/watchOS 11/visionOS 2 — below .v26, no `nonisolated(unsafe)` globals, per [[nebula-swift6-concurrency]]).\n- Sendable value types with derived/conditional conformance (`AttributedString`, `Regex<Output>` → `NebulaRegex<Output>`, `NebulaStringLocalization`, `NebulaStringDetectedEntity`) — no `@unchecked Sendable` on Nebula types. (`LocalizedStringResource` is Sendable at iOS 18, below .v26, so any wrapper storing it is Sendable on the Nebula floor.)\n- Above-floor OS-introduced APIs gated with `@available(iOS 26.4, *)` (etc.) and documented \"since Nebula 26.4\" (base64URLAlphabet, omitPaddingCharacter, ianaName getter AND init?(ianaName:) initializer).\n- Foundation scope only for `AttributedString` (no `import UIKit`, no SwiftUI scopes) — keeps the foundation UIKit-free and identical across iOS/macOS/tvOS/watchOS/visionOS. Verified Foundation ships `AttributeScopes.FoundationAttributes` usable with `import Foundation` only.\n- DocC on every public symbol; deprecation via `@available(*, deprecated, message:)`.\n- Single SPM target `Nebula` with internal `Extensions/String|Regex|Types` folders — mirrors Cosmos single-target architecture (see [[nebula-spm-architecture]]).\n\n## Risks & open questions\n\n- `NSDataDetector` is ObjC (not in the textual `.swiftinterface`; verified, grep count 0); source of truth is `NSRegularExpression.h` `API_AVAILABLE(macos(10.7), ios(4.0), watchos(2.0), tvos(9.0))`. Its `enumerateMatches` block uses `UnsafeMutablePointer<ObjCBool>` + `NSRange`; prefer the array-returning `matches(in:options:range:)` overload and cache the detector in `Mutex<NSDataDetector?>` (Synchronization, iOS 18+ — below .v26, no `nonisolated(unsafe)` globals, per [[nebula-swift6-concurrency]]).\n- `NebulaRegex<Output>` Sendable conformance must be conditional on `Output: Sendable` — a blanket conformance would be unsound. (`RegexComponent` the protocol is not Sendable-declared, but that is irrelevant.)\n- `NebulaRegexPatterns` deliberately excludes email/phone regex (Apple: don't validate data with regex/NSDataDetector). Document this in DocC.\n- Hex codec is hand-rolled; handle odd-length strings (failable init), upper/lowercase, optional `0x` prefix; unit-test `Data ⇄ hex ⇄ Data` round-trip.\n- Four above-floor APIs (base64URLAlphabet, omitPaddingCharacter, ianaName getter, AND init?(ianaName:) — all @available(macOS 26.4, iOS 26.4, tvOS 26.4, watchOS 26.4, visionOS 26.4, *)) will not compile on the `.v26` baseline if unconditional — must be separate `@available(iOS 26.4, *)` overloads. (Original research listed only the ianaName getter; the initializer is also 26.4.)\n- `NebulaStringLocalization` should own the `.lproj` + `.xcstrings` fallback resolution (from `CosmosLocalizationConfiguration`), not rely solely on `String(localized:bundle:locale:)`.\n- Open: should `NebulaRegex` expose `init?(pattern: String)` runtime compilation (reDoS risk — default no)? Should `NebulaStringDetectedEntity` carry `Range<String.Index>` for UI highlighting? Does base64/hex codec belong on `String` here or on `Data` (see [[nebula-data-url-extensions]])? watchOS 9 baseline behavior for `String(localized:)` to verify before locking the contract. Should Nebula expose `String.Encoding.init?(ianaName:)` alongside the getter (both 26.4)?\n\n## Sources\n\n- [Foundation.swiftinterface (local file, Xcode 27 Beta 3 /
